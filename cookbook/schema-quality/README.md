# Schema-Quality Cookbook

How schema design drives extraction accuracy and stability in Retab — proven
with a measured experiment, not opinion.

## The experiment

Five documents of the **same type** (invoices), processed by **one schema**.
They differ only in *form* — which optional fields are printed:

| Field | full | no_name | no_code | minimal | mixed |
|---|:--:|:--:|:--:|:--:|:--:|
| customer_name | ✓ | · | ✓ | · | ✓ |
| customer_code | ✓ | ✓ | · | · | ✓ |
| purchase_order | ✓ | ✓ | · | · | · |
| discount | ✓ | ✓ | ✓ | · | · |
| due_date | ✓ | ✓ | ✓ | · | ✓ |
| tax | ✓ | ✓ | ✓ | · | ✓ |

(✓ present, · absent). This is the right shape to test **nullable types**: the
same field is present on some invoices and genuinely absent on others. The
corpus also varies *convention*: `discount` is printed as a negative number,
`invoice_date` uses European `DD/MM/YYYY` on several invoices (ground truth is
always ISO 8601), and `currency` is printed in verbose forms (`Euros`, `US
Dollars`, `Pounds Sterling`, `€`) while ground truth is the canonical ISO-4217
code (`EUR`/`USD`/`GBP`) — so the levers are tested on more than one field.

We compare four schema variants, each adding exactly one lever on top of the
previous, so every difference is attributable:

1. **baseline** — generated by `retab schemas generate` from one representative
   invoice. The agent marks **every field `required`** with **no nullable
   types** and **no reasoning prompts** (see `schemas/baseline.json`).
2. **nullable** — the baseline with the five optional fields retyped
   `["<type>", "null"]`. They stay `required`: under strict structured output
   every property is emitted anyway, so optionality is carried by the null type,
   not by dropping the key from `required`.
3. **reasoning** — the nullable schema plus an **`X-ReasoningPrompt`** on each
   optional field (and an `X-SystemPrompt`).
4. **enum** — the reasoning schema plus an **`enum`** of ISO-4217 codes on
   `currency`, normalizing the printed vocabulary to a canonical code.

### What is a reasoning prompt?

`X-ReasoningPrompt` is a Retab JSON-Schema extension. Its text is given to the
model as a private, per-field instruction — a "think about it this way before
you answer" hint. It never appears in the output; it only steers extraction.
`X-SystemPrompt` is the same idea applied to the whole schema. In this
experiment the reasoning prompts say, per field, *"only return this value if it
is actually printed; if it is absent return null, and don't copy another
field"* — and for `discount`, *"use the printed value, which is typically
negative."*

## Results

Latest run (model `retab-micro`, `n_consensus=5`) is in
[`RESULTS.md`](RESULTS.md). Headline:

| Variant | Overall | Present-field | **Absent-field** | Currency |
|---|---:|---:|---:|---:|
| baseline | 83% | 86% | **73%** | 3/5 |
| nullable | 90% | 88% | **100%** | 2/5 |
| reasoning | 95% | 94% | **100%** | 2/5 |
| enum | **98%** | **98%** | **100%** | **5/5** |

### Interpretation

Three clean lever wins, plus one honest caveat about run-to-run variance:

- **nullable → absence.** On the invoices with no discount line (and the one
  with no tax line) the baseline returned `0`, because a `required` non-nullable
  number has no way to say "not present." Making those fields nullable fixed it
  (`null`): absent-field accuracy **73% → 100%**. *Lesson: mark genuinely-optional
  fields nullable; a required, non-nullable field forces a fabricated value.
  Keeping the field required but nullable is enough — and is the right shape for
  strict structured output.*
  - Note: the baseline returned `null` correctly for the absent *string* fields
    (customer_name, code, PO) even though they were required — modern models
    often leave absent strings empty. The reliable failure was on *numbers*,
    where `0` looks like a legitimate value.
- **reasoning → sign convention.** The discount is printed negative; where it
  was present, baseline and nullable returned the **wrong sign** (`150`, `48`,
  `32.25` instead of `-150`, `-48`, `-32.25`) on all three invoices. The
  `X-ReasoningPrompt` stating the printed value is negative flipped the sign on
  all three. Present-field accuracy **88% → 94%**. *Lesson: a reasoning prompt
  encodes a convention a bare type cannot — but watch the likelihood: in the
  enum run one discount came back positive at confidence `0.60` (not `1.00`),
  the model's own "I'm unsure" signal, a reminder that a small model mitigates
  rather than guarantees.*
- **enum → vocabulary normalization.** `currency` is printed verbosely (`Euros`,
  `US Dollars`, `Pounds Sterling`, `€`). A free-text field echoes the printed
  form, so baseline / nullable / reasoning scored only **2–3 / 5** against the
  canonical ISO code. Adding an `enum` of ISO-4217 codes normalized every value
  (`Euros → EUR`, `US Dollars → USD`): **5 / 5**, lifting present-field accuracy
  **94% → 98%**. *Lesson: for a categorical field, an enum pins the output to a
  fixed vocabulary instead of whatever surface form the document happens to use.*
- **Caveat — the European date risk is real but noisy.** The baseline misread
  two `DD/MM/YYYY` dates this run (`03/02/2024 → 2024-03-02`, `10/02/2024 →
  2024-10-02`, swapping day and month); the other variants returned the correct
  ISO date. But the correction already appears at **nullable**, which has no
  date mechanism — so this is partly sampling variance, not a clean lever effect.
  The reasoning schema does carry a date prompt for exactly this pattern.
  *Lesson: a single run can over- or under-state a "risky-looking" pattern —
  measure across runs before drawing a conclusion.*

So the levers are complementary and stack cleanly: **nullable** fixes *absence*,
**reasoning** fixes the *sign convention*, **enum** fixes *vocabulary* — overall
accuracy climbs **83% → 90% → 95% → 98%**. Likelihood (consensus agreement)
stayed high throughout — often `1.00` even when the baseline was confidently
wrong about the discount sign: **high confidence does not imply correctness.**
Only a handful of fields fell below `0.90` (mostly the verbose currency forms,
plus the enum-run discount at `0.60`; full list in [`RESULTS.md`](RESULTS.md)
§5), and likelihood barely moved while accuracy climbed 15 points — so it could
not have ranked the schemas. Accuracy against ground truth is the metric that
matters.

## How to run

```bash
pip install reportlab
python generate_documents.py          # build the 5 invoice PDFs + manifest
python -m harness.variants            # derive nullable.json + reasoning.json + enum.json from baseline.json
python -m harness.experiment          # extract, score, write RESULTS.md (cached; re-runs are free)
python -m harness.lint schemas/*.json # static schema linter (no API calls, free)
```

A narrated, runnable walkthrough lives in [`schema-quality.ipynb`](schema-quality.ipynb).

To regenerate the baseline itself from the agent:

```bash
retab schemas generate --file documents/invoice_full.pdf --model retab-micro \
  --format schema --output json > schemas/baseline.json
```

## File guide

| Path | Role |
|---|---|
| `generate_documents.py` | Builds the 5 invoice PDFs deterministically + the ground-truth manifest. |
| `documents/*.pdf` | The corpus. |
| `documents/manifest.json` | Ground truth: correct value of every field (null = absent). |
| `schemas/baseline.json` | The agent-generated schema (the baseline). |
| `schemas/nullable.json`, `reasoning.json`, `enum.json` | Derived variants (built by `harness/variants.py`). |
| `schemas/field_map.json` | Maps canonical field names → the schema's property names. |
| `harness/variants.py` | Derives nullable, reasoning & enum from the baseline. |
| `harness/runner.py` | Runs one extraction via the Retab CLI; caches results. |
| `harness/scoring.py` | Typed/normalized scoring of output vs ground truth + likelihoods. |
| `harness/report.py` | Renders the Markdown report. |
| `harness/experiment.py` | Orchestrates docs × variants → `RESULTS.md`. |
| `harness/lint.py` | Static schema linter encoding the proven patterns. |
| `schema-quality.ipynb` | Narrated, runnable walkthrough of the whole experiment. |
| `RESULTS.md` | Latest experiment output. |
